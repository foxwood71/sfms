# 📘 SFMS Phase 1 SD - 02 IAM 및 USR 상세 설계서

* **문서 버전:** v1.5 (Dynamic Authorization & Security Audit)
* **최종 수정일:** 2026-03-21
* **주요 내용:** 계정 관리 로직, 동적 RBAC 시스템, 보안 감사 정책

---

## 🔐 1. 인증 및 권한 (IAM) 설계

### 1.1 동적 슈퍼유저 (Dynamic Superuser) 로직
시스템은 특정 역할 코드에 종속되지 않는 유연한 권한 체계를 가집니다.
*   **작동 방식**: 백엔드 `User.is_superuser` 프로퍼티가 사용자의 모든 역할을 순회하며 `permissions` 데이터 확인.
*   **핵심 키**: `{"ALL": ["*"]}` 또는 `{"all": ["*"]}` 설정이 포함된 역할 보유 시 즉시 슈퍼유저로 판별.
*   **영향 범위**: 메뉴 필터링 우회, 전체 도메인 관리자 권한(`check_domain_admin`) 자동 통과.

### 1.2 보안 감사 (Security Auditing)
로그인 및 세션 관련 모든 위협 시도를 추적합니다.
*   **LOGIN_FAILURE**: 비밀번호 불일치뿐만 아니라, 존재하지 않는 ID로의 접근 시도도 모두 기록.
*   **ACCOUNT_LOCKED**: 5회 연속 실패 시 계정 잠금 이벤트를 별도로 기록하여 관리자 모니터링 지원.
*   **정보 수집**: 모든 보안 로그에는 요청자의 IP 주소와 User-Agent 정보를 강제 포함.

## 👤 2. 사용자 및 조직 (USR) 설계

### 2.1 조직도 트리 (Organization Tree)
*   **루트 노드 표준**: 모든 도메인의 조직도 트리는 `"전체 조직도"`를 최상단 가상 노드로 가집니다.
*   **아이콘 표준**: 루트 노드에는 반드시 **`ApartmentOutlined`** 아이콘을 사용하여 계층 구조임을 명시합니다.
*   **비활성 처리**: 퇴사자나 폐쇄된 부서는 취소선(`line-through`) 및 회색 처리를 통해 시각적으로 구분합니다.

### 2.2 사용자 데이터 연동
*   **사번 체계**: 영문 대문자와 숫자의 조합으로 구성하며, 공백 없이 엄격한 유효성 검사를 거칩니다.
*   **메타데이터 확장**: 직위(`pos`), 직책(`duty`) 등 변동성이 높은 인사 정보는 `JSONB` 컬럼(`metadata`)에 저장하여 스키마 변경 없이 유연하게 대응합니다.
